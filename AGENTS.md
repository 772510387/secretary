# Secretary Agent 协作规则

本文件给后续所有 AI coding agent 使用。任何实现前先读本文件，再读目标模块的 `README.md`。

## 项目目标

`secretary` 是一个独立 A 股智能交易辅助系统。它吸收 OpenClaw 式强智能体交易体验和 TradingAgents-CN 的研究流程思想，但保持本项目独立实现与可审计演进。

## 核心边界

- 确定性规则必须写成代码：交易规则、风控、仓位、T+1、日志、记忆写入权限。
- LLM 可以承担强判断与强策略职责：新闻解释、策略推演、标的排序、交易意图、复盘、自然语言报告和执行提案。
- 账户、订单、规则文件和长期记忆的最终状态必须来自显式工具、执行适配器、账本回执或人工确认记录，不能靠无审计文本覆盖。
- 运行模式由配置和阶段决定；模拟盘、人工确认、只读 broker、实盘 broker 都必须有清晰入口、权限、审计和回滚线索。

## 开发前上下文加载顺序

2. 读 `docs/architecture/README.md` 了解总体架构。

6. 只在确有必要时回读 `docs/requirements` 原始需求。

## 实现纪律

- TypeScript 优先。
- 领域层不得直接调用网络、文件系统、模型 SDK、券商 SDK。
- 外部依赖统一放入 `src/infrastructure`。
- 入口层只负责解析请求、鉴权、返回结果，不写交易规则。
- 所有资金、持仓、订单、风控、模型建议都必须可审计。
- 数据写入要使用原子写入和备份策略。
- 配置和密钥分离。密钥只来自环境变量或本机密钥管理，不进入仓库。

## 测试要求

任何涉及以下内容的改动必须补测试：

- 金额、数量、仓位、成本价、盈亏计算。
- T+1、100 股整数、主板过滤、涨跌停、单股上限。
- 止损、熔断、禁买、重复下单防护。
- 记忆写入、审计日志、异常恢复。
- Provider 或 Broker 适配器的错误处理。

## 提交前检查

- 目标模块 README 是否仍然准确。
- 是否保留了强模型判断到执行适配器的清晰边界和审计回执。
- 是否新增密钥、账号、交易参数且未按配置/密钥分离管理。
- 是否保留了审计线索。
- 是否新增了测试或说明了不能测试的原因。
