# Secretary Agent 协作规则

本文件给后续所有 AI coding agent 使用。任何实现前先读本文件，再读目标模块的 `README.md`。

## 项目目标

`secretary` 是一个独立 A 股智能交易辅助系统。它从 OpenClaw 原型经验中吸收设计，但不依赖 OpenClaw；它可以吸收 TradingAgents-CN 的研究流程思想，但不能把 TradingAgents-CN 的专有 `app/`、`frontend/` 复制进本项目。

## 核心边界

- 确定性规则必须写成代码：交易规则、风控、仓位、T+1、日志、记忆写入权限。
- LLM 只负责模糊判断：新闻解释、策略推演、复盘、自然语言报告。
- LLM 不允许直接改账户、直接发单、直接覆盖规则文件。
- 真实交易默认关闭。除非需求明确进入实盘阶段，否则所有实现都必须以模拟盘为默认路径。

## 开发前上下文加载顺序

1. 读 `README.md` 了解项目边界。
2. 读 `docs/architecture/README.md` 了解总体架构。
3. 读 `docs/architecture/module-map.md` 找到目标模块。
4. 读目标目录下 `README.md`。
5. 读相关测试目录 `README.md`。
6. 只在确有必要时回读 `docs/requirements` 原始需求。

## 实现纪律

- TypeScript 优先。
- 领域层不得直接调用网络、文件系统、模型 SDK、券商 SDK。
- 外部依赖统一放入 `src/infrastructure`。
- 入口层只负责解析请求、鉴权、返回结果，不写交易规则。
- 所有资金、持仓、订单、风控、模型建议都必须可审计。
- 数据写入要使用原子写入和备份策略。
- 配置和密钥分离。密钥只来自环境变量或本机密钥管理，不进入仓库。

## 测试要求

任何涉及以下内容的改动必须补测试：

- 金额、数量、仓位、成本价、盈亏计算。
- T+1、100 股整数、主板过滤、涨跌停、单股上限。
- 止损、熔断、禁买、重复下单防护。
- 记忆写入、审计日志、异常恢复。
- Provider 或 Broker 适配器的错误处理。

## 提交前检查

- 目标模块 README 是否仍然准确。
- 是否误把 LLM 放进确定性决策链。
- 是否新增密钥、账号、真实交易参数。
- 是否保留了审计线索。
- 是否新增了测试或说明了不能测试的原因。
