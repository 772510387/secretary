# 向量语义记忆检索评估

## 背景

R7 需要评估是否在 `MemoryRegistry` 之外增加语义检索能力。当前系统已经具备文件系统索引、关键词检索、时间过滤和最近记忆 metadata 读取。记忆中可能包含研究观点、复盘结论、审计摘要和规则文本，任何增强检索都必须继续满足不泄漏密钥、不读取账户敏感正文、不让模型直接改规则或下单的边界。

## 决策

第一阶段继续保留关键词检索和文件索引作为默认路径，不引入大型向量数据库，也不在运行态默认调用远程 embedding 服务。

后续如果要做语义检索，优先采用可重建的旁路索引：

- 原始记忆仍保存在 `memory/` 的 Markdown/JSON/JSONL 文件中。
- 向量索引只保存脱敏摘要、文档路径、更新时间、内容 hash、embedding 模型版本和向量。
- 索引可删除、可全量重建，不作为事实源。
- 检索结果仍只返回路径、摘要、更新时间和 metadata，不返回完整敏感正文。

## 方案比较

本地 embedding：

- 优点：隐私边界最好，不把记忆正文发给外部服务；费用稳定。
- 缺点：需要本机模型、运行依赖和性能预算；中文金融语义质量需要单独评估。
- 适用：长期默认候选，但必须先做离线 benchmark 和资源上限。

远程 embedding：

- 优点：质量和维护成本较好，接入快。
- 缺点：有费用、限流、网络失败和隐私外发风险；必须脱敏、摘要化并显式启用。
- 适用：只适合可外发的低敏摘要，不适合账户、订单、密钥、完整审计正文。

SQLite/文件索引：

- 优点：轻量、可审计、易备份；适合保存路径、hash、mtime、关键词、摘要和未来向量数组。
- 缺点：向量相似度能力有限，规模变大后需要额外优化。
- 适用：第二阶段优先方案，可先做 SQLite FTS，再评估向量列或外部 ANN。

大型向量库：

- 优点：召回能力和扩展性强。
- 缺点：引入运维复杂度、数据迁移、隐私边界和失败模式；当前项目规模不需要。
- 适用：暂缓，除非记忆规模和召回质量证明关键词/SQLite 方案不足。

## 隐私和安全

- 不索引 `portfolio/`、`config/`、`secrets/`、`broker/`、`orders/`。
- 不索引完整审计正文、完整研究正文、完整报告正文或提案 rationale。
- 入索引内容必须先脱敏 `apiKey`、`token`、`secret`、`password`、`account`、broker 账号等字段。
- LLM 或 embedding provider 只用于生成向量或解释检索结果，不能修改记忆、规则、账户或订单。

## 费用和限流

- 远程 embedding 必须有每日请求数、每日费用、单文档最大字符数和重试上限。
- 请求必须批量化并缓存，缓存键至少包含文档 hash、模型名、模型版本和脱敏策略版本。
- 429、5xx、超时或坏响应时降级到关键词检索，不阻塞核心风控、审计和交易边界。

## 缓存和重建

- 索引记录包含 `relativePath`、`updatedAt`、`contentHash`、`summary`、`embeddingModel`、`embeddingVersion`、`redactionVersion`。
- 文档 hash 或脱敏策略变化时重建对应记录。
- 模型升级时允许全量重建；旧索引可保留到新索引构建完成后原子切换。
- 索引损坏时直接删除重建，不能影响原始 `memory/` 文件。

## 失败降级

- embedding 不可用时，`MemoryRegistry.search()` 继续走关键词检索。
- 向量索引缺失或过期时，只降低语义召回，不影响确定性规则、风控或审计。
- 检索结果必须标注来源模式：`keyword`、`semantic` 或 `hybrid`。

## 后续动作

- 保持 R7 第一阶段关键词检索和时间过滤。
- 在 R8/R9 之后，如确有需要，再设计 `SemanticMemoryIndex` 接口和离线重建脚本。
- 先用 SQLite/文件索引做最小 proof-of-concept，再决定是否需要远程 embedding 或向量库。
