# 协调 + 实施会话 · 开场 prompt

> 用法：在你想用来推进实现的那个会话里，直接说「读 docs/stock-strategy/_MERGE-PROMPT.md 并执行」。

你是这个项目（secretary）的**协调 + 实施会话**。各探查会话的结论已落在 `docs/stock-strategy/` 下，但它们之间**互不共享上下文**——你只能从这些文件重建全貌。按以下步骤来，**不要跳过第三步直接写代码**。

## 第一步：读齐
读 `docs/stock-strategy/` 下全部 `.md`（所有发现文件 + README 索引 + _TEMPLATE）。逐份提取每份的：第 3 节现状判定、第 4 节待办、第 5 节信息缺口、第 6 节触碰文件清单。

## 第二步：合并 + 建冲突地图
- 把所有「待办」汇成一张去重后的总表；合并相邻/重叠主题（例如 `premarket-915-indicators` 与 `pre-market-brief`）。
- 用各份第 6 节的「触碰文件清单」建**冲突矩阵**：改同一文件的任务 → 必须串行；文件全新/不相交的任务 → 可并行（worktree）。
- 标出**共改热点**：`src/app/brain-agent-tools.ts`、`src/domain/market/index.ts`、`src/infrastructure/providers/index.ts`(barrel)。这些放到最后一个单独的「收口」步，不要并行。

## 第三步：把要拍板的决策先摆出来，别擅自开工
至少下面这些需要先让我（用户）定，带上你的推荐选项问我：
- `strategy-knowledge-base`：桥接方案 A vs B（该文件标了待拍板）。
- 一字板 / 封单 的确定性计算挂在 9:15(`call_auction_watch`) 还是 9:25(`pre_open_confirmation`) 节点。
- `review-grounding` 指出的「复盘数字/时区/理由全靠模型编」——确定性事实层的边界划到哪。
- 各份里标注的**疑似真 bug**（如 T+1 rollover 持仓卖不出）——**先验证能否复现**，再决定是否插队成 P0。
把这些整理成清单问我，我答完你再排期。

## 第四步：产出单一行动计划
输出：优先级排序的任务列表 + 每个任务的 worktree/分支建议 + 最后的串行「收口」步骤 + 每块的验收方式（跑哪些测试）。**先给我看计划，我确认后再动代码。**

## 第五步：实施
按计划做。遵守项目既有硬约束：仅主板、不接真实券商/不真下单、确定性（眼/小脑）层**不许模型编数**、改动要有测试覆盖。每完成一块跑一次测试再继续。
